# 中医大众学习项目方向规划

## 1. 项目目标

把《圆运动的古中医学》和《火神门系列医易互通》的核心思想，转化成普通用户也能理解、记住、反复学习的交互式学习产品。

这个项目不应该先被定义成“做一个 3D 游戏”，而应该先定义成：

**一个以中医概念可视化、动态化、交互化为核心的学习产品。**

3D、动画、游戏化只是手段，不是起点。

## 2. 推荐方向

### 最推荐的产品形态

先做一个 **Web 优先的交互式学习平台**，核心由三部分组成：

1. **动态课程**
   把两本书拆成一节节短内容，每节只讲一个核心概念。

2. **图表交互化**
   把书里的图表重绘成可点击、可播放、可切换层级的动态图。

3. **轻度游戏化练习**
   用“判断气机方向”“选择治疗思路”“拖动药法归类”等形式做练习，而不是直接做大型 RPG。

### 不建议一开始就做什么

不建议第一阶段直接做：

- 大型 3D 世界
- 开放式剧情游戏
- 重资产角色系统
- 复杂联网社区

原因很简单：当前最大的不确定性不在技术，而在 **内容如何拆解成能教会人的结构**。

## 3. 先做成什么

### 方案 A：交互式中医学习站

这是最稳的起点，也是我最推荐的。

它的结构可以是：

- 首页：这两本书的学习地图
- 模块页：圆运动、扶阳、坎离水火、桂枝法、附子法、脉法、舌象、医案
- 图表页：把所有关键图表做成动画图谱
- 练习页：用小测验和病例推演检验理解
- 进度页：记录学到哪里、哪些概念没掌握

### 方案 B：2.5D 叙事式课程

如果你特别想强调“会动”和“沉浸感”，可以做成章节式体验：

- 每章像一个短场景
- 用镜头运动、粒子流动、器官/五行关系动画讲概念
- 用户点击节点、切换视角、推进讲解

这更像“会动的知识展览馆”，比“游戏”更适合第一版。

### 方案 C：中后期再加入模拟式玩法

等内容模型稳定后，再做：

- “气机调度”模拟器
- “辨证思路”闯关
- “方药归类”卡牌式练习
- “病例推演”分支互动

## 4. 内容应该如何拆

这类书不能按 PDF 页码去做产品，要按“知识原子”拆。

### 建议的最小内容单位

每个知识点都拆成一个卡片对象，至少包含：

- `id`
- `title`
- `source_book`
- `chapter`
- `summary`
- `plain_explanation`
- `keywords`
- `relations`
- `visual_type`
- `quiz_type`
- `warning`

### 推荐的内容类型

1. **概念卡**
   例如：中气、升降出入、圆运动、坎离、少阴、厥阴。

2. **关系卡**
   例如：肝木如何影响脾土，桂枝法为什么是“开”，附子法为什么是“归”。

3. **方法卡**
   例如：桂枝法、附子法、茯神法、右降法。

4. **图表卡**
   例如：五行运行图、坎离水火图、六经层次图、圆运动示意图。

5. **病例卡**
   例如：某个医案中的脉象、判断逻辑、先后手。

6. **误区卡**
   例如：“扶阳不等于无脑用热药”“中医不是先背病名”。

## 5. 哪些内容适合 3D，哪些不适合

### 适合 3D 的内容

- 气机升降出入
- 圆运动整体流转
- 坎离水火上下交通
- 六经层次与空间感
- 上热下寒、内外不通这类结构型概念

### 不适合强做 3D 的内容

- 大段理论原文
- 药味分类表
- 医案条文阅读
- 长篇解释文字

### 对图表的建议

书里的图表不要直接拿扫描图做动画，应该：

1. 重新提炼含义
2. 重绘成 SVG 或分层矢量图
3. 把每一层做成可单独高亮和播放的动画

也就是说，**图表要被“重构”，不是被“搬运”。**

## 6. 推荐的产品架构

建议做成四层架构。

### 第一层：内容源层

保存原始资料和整理结果：

- 原始 PDF
- OCR 文本
- 人工校对文本
- 摘要文档
- 图表清单

这一层的重点是“可追溯”，以后每个学习模块都能回到来源。

### 第二层：知识结构层

把内容整理成结构化数据，而不是散落在长文里。

推荐结构：

- `books`
- `chapters`
- `concepts`
- `methods`
- `diagrams`
- `cases`
- `quizzes`
- `glossary`

这层本质上就是项目的大脑。

### 第三层：表现层

把结构化内容输出成不同学习体验：

- 文章式课程
- 动态图表
- 2D/2.5D 动画
- 3D 场景
- 测验与互动

### 第四层：学习引擎层

控制用户如何学：

- 学习路径
- 难度递进
- 复习提醒
- 错题回看
- 概念关联跳转

## 7. 推荐技术栈

### 前端

推荐：

- `Next.js`
- `React`
- `TypeScript`

原因：

- 适合内容站和交互应用混合
- 容易做 SEO、静态生成、课程页
- 后面加用户系统和后台也方便

### 2D 动画

推荐优先级：

1. `SVG + Motion`
2. `Rive`
3. `Lottie`

理由：

- 图表动画大多数不需要 3D
- 2D 动画成本更低、表达更清楚
- 更适合移动端和网页加载

### 3D

只在必要模块使用：

- `react-three-fiber`
- `three.js`
- `drei`

适合做：

- 圆运动空间演示
- 人体气机流动
- 经络/器官关系的沉浸场景

### 内容存储

第一阶段不要一上来就上重数据库。

推荐：

- `MDX` 存课程正文
- `JSON` 或 `YAML` 存结构化知识卡
- 本地脚本生成索引

等内容变多、需要多人协作时，再升级到：

- `PostgreSQL`
- 轻后台 CMS

### 搜索与索引

第一阶段：

- 本地全文索引
- 标签过滤

第二阶段：

- `Typesense` 或 `Meilisearch`

## 9. 先做哪三个核心模块

如果只能先做三个，我建议是：

### 模块 1：圆运动基础

目标：让普通用户第一次理解“中医看的不是病名，而是气机方向”。

表现形式：

- 动态总览图
- 升降出入动画
- 五脏功能关系图
- 小测验

### 模块 2：扶阳与坎离水火

目标：让用户看懂《火神门》的主线，不再觉得玄。

表现形式：

- 坎离交通动画
- 桂枝法 vs 附子法 对比图
- 上热下寒、内外不通的动态场景

### 模块 3：图表交互图谱

目标：把书中最难啃的图表，变成可点、可看、可播放的学习入口。

表现形式：

- 每张图单独一页
- 分层开关
- 术语解释
- 动画播放
- 关联章节跳转

## 10. 建议的开发路线

### 第一阶段：做内容底盘

时间：1 到 2 周

任务：

- 清理 OCR 文本
- 统一术语
- 建立知识卡 schema
- 整理图表目录
- 为每张图写“这张图到底要教什么”

产出：

- 可维护的结构化内容库

### 第二阶段：做 MVP

时间：2 到 4 周

任务：

- 搭建 Web 应用
- 实现课程页面
- 实现动态图表组件
- 做 1 到 2 个核心动画场景
- 做基础练习系统

产出：

- 一个能实际体验、能给别人试学的版本

### 第三阶段：加入选择性的 3D

时间：2 到 4 周

任务：

- 给最有空间感的概念做 3D
- 做镜头切换和交互
- 优化性能和移动端表现

产出：

- 有明显辨识度的沉浸式模块

### 第四阶段：做病例推演和游戏化

时间：后续迭代

任务：

- 加入分支互动
- 做辨证练习
- 加学习进度和复习机制

## 11. 我对这个项目的核心判断

这个项目真正难的地方不是“怎么做动画”，而是：

**如何把流派化、抽象化、带有强解释框架的中医内容，拆成大众能吸收的学习路径。**

所以最合理的顺序是：

1. 先把知识结构化
2. 再把关键图表动态化
3. 再把少数高价值概念做成 3D
4. 最后才考虑更重的游戏化

## 12. 当前最建议的结论

如果你现在就要定方向，我建议定成：

**“一个以动态图表 + 交互课程 + 少量高质量 3D 场景为核心的中医大众学习平台。”**

这比“直接做 3D 游戏”更稳，也更容易先做出成品。

## 13. 下一步最值得立即做的事

接下来最值得做的不是建 3D，而是先完成下面三件事：

1. 列出两本书最值得转化的 20 到 30 个核心知识点
2. 列出所有值得重绘的图表
3. 选 1 个模块先做 MVP 样例

我建议第一个 MVP 模块就做：

**“圆运动 + 中气 + 升降出入”**

因为这部分最适合做成大众入门，也最适合动画化。
