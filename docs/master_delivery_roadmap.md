# 项目总路线：从当前原型走到可交付成品

## 1. 终局定义

这个项目的终局，不应该定义成“做完几个会动的页面”，而应该定义成：

**一个能让普通用户系统学习《圆运动的古中医学》与《火神门系列医易互通》核心思想的交互式学习产品。**

它至少要满足四个结果：

1. 用户能按路径学，而不是只能随便点。
2. 两本书的核心内容已经被结构化，不再散落在 PDF 和 OCR 里。
3. 关键图表已经从静态书页变成动态图和练习入口。
4. 产品已经具备最基本的学习闭环：看概念、看图、做题、回看错因。

## 2. 最终交付形态

项目结束时，建议至少交付下面 6 个层面：

1. 内容层
   两本书的核心章节、概念、方法、图表、病例、练习已经结构化入库。

2. 课程层
   至少 6 个模块都具备完整课程脚本，而不是只有零散概念卡。

3. 图表层
   至少 10 到 15 张高价值图表完成重绘，其中 5 到 8 张做成高质量交互图。

4. 练习层
   至少 3 类练习已经成型：概念判断、动作分流、病例推演。

5. 产品层
   Web 版具备首页、模块页、图表页、练习页、进度页。

6. 演示层
   至少 1 到 2 个高辨识度动画或 3D 场景，能代表项目视觉高度。

## 3. 当前状态

当前已经完成的是：

1. 两本书资料已经入库，PDF 和 OCR 都在仓库中。
2. 已经抽出 25 个核心概念、6 个模块、15 张优先图表。
3. 已经有多个交互原型：
   圆运动、坎离交通、桂枝开门、附子归根、茯神缓转、双图对照、三动作总览、病例分流练习。
4. 已经有 Next.js Web 骨架和基础内容 schema。
5. 已经开始有题库，并且题库已经从组件里抽到 `content/practices/`。

当前还没完成的是：

1. 两本书的完整覆盖地图。
2. 4 个模块的完整课程脚本。
3. 系统化的图表重绘资产流程。
4. 错题回看、学习进度、复习机制。
5. 完整病例推演。
6. 明确的上线版和展示版边界。

## 4. 总体原则

整个项目后面都按这 5 条原则推进：

1. 先做内容结构，再做炫技表现。
2. 优先做学习闭环，不优先做重型 3D。
3. 只把最适合空间表达的概念做成 3D。
4. 所有页面尽量内容驱动，不要把内容继续写死在组件里。
5. 每一个阶段都必须产出可演示版本，而不是只做底层准备。

## 5. 大阶段路线

### 阶段 A：内容覆盖与课程底盘

目标：
把“两本书到底覆盖到哪里”讲清楚，并把剩余模块补到可以继续开发的状态。

必须完成的事：

1. 建一份覆盖地图
   原书章节 -> 模块 -> 概念 -> 图表 -> 练习。

2. 补齐术语表
   统一“中气、升降出入、扶阳、开门、归根、缓转、坎离”等核心词。

3. 补齐 4 个未完整展开的模块脚本
   当前完整脚本只有 `foundations` 和 `fu-yang-core`。

4. 给每个模块补“学习问题、课程节、练习想法、图表入口、下一模块”。

5. 给每个练习题补“关联概念、关联图表、难度、错因类型”。

阶段产出：

1. 内容覆盖地图文档
2. 统一术语表
3. 所有模块的课程脚本 JSON
4. 第一版完整题库 schema

验收标准：

1. 6 个模块都有完整 `module-details`
2. 每个模块至少有 3 到 5 节课程
3. 每个模块至少有 1 种练习入口
4. 两本书的核心章节都能在覆盖地图里找到归属

### 阶段 B：MVP 学习站成型

目标：
把当前原型收成一个能给真实用户试学的完整 MVP。

必须完成的事：

1. 首页升级
   不是只列模块，而是给出推荐学习顺序、预计时长、起步建议。

2. 模块页升级
   每个模块页形成固定结构：
   模块介绍 -> 课程节 -> 图表入口 -> 小练习 -> 下一步。

3. 图表页整理
   把关键图表做成统一目录页，不再只靠零散 prototype 链接。

4. 练习系统升级
   支持答题状态、得分、错题回看、推荐回到哪张图。

5. 学习进度页
   最低限度先做本地存储版，不需要先上账号系统。

6. 页面体验统一
   统一视觉风格、按钮语义、导航结构和页面节奏。

阶段产出：

1. 第一版完整学习站
2. 6 个模块页
3. 图表目录页
4. 练习页与进度页

验收标准：

1. 新用户可以从首页进入并完成一条完整学习链
2. 做错题后能被正确带回相应原型或课程
3. 至少有 1 个模块能完整体验“看课 -> 看图 -> 做题 -> 回看”

### 阶段 C：图表资产体系化

目标：
让图表开发不再是临时画几张，而是形成稳定生产流程。

必须完成的事：

1. 建图表台账
   每张图记录来源、用途、重绘状态、动画状态、页面挂载位置。

2. 建图表分层规范
   哪些元素是静态底图，哪些是高亮层，哪些是运动层，哪些是注释层。

3. 明确输出格式
   `svg-motion`、`rive`、`three-scene` 分别适用什么场景。

4. 完成 10 到 15 张图的重绘计划
   其中优先完成 5 到 8 张真正上线使用的图。

5. 给每张重点图补“这张图教什么、用户容易误解什么、练习如何接入”。

阶段产出：

1. 图表台账
2. 图表重绘模板
3. 图表页面目录
4. 重点图统一接入 Web

验收标准：

1. 图表不是孤立存在，而是都能回到课程和练习
2. 新做一张图时，团队知道该怎么从原书推进到上线页

### 阶段 D：练习系统与学习引擎

目标：
让这个产品从“能看”升级到“能学会”。

必须完成的事：

1. 题库分层
   基础题、进阶题、病例题分开。

2. 错题系统
   记录错因，不只是记录对错。

3. 复习机制
   已学概念、薄弱动作、待复习图表要能追踪。

4. 学习路径推荐
   用户当前卡在哪个动作，就推荐回看哪张图、哪个模块。

5. 进度记录
   先本地，再考虑是否接账号。

阶段产出：

1. 第一版学习引擎
2. 本地进度和错题记录
3. 错题回看链路

验收标准：

1. 用户重复使用产品时，有累计感
2. 做题不是一次性体验，而是可复习的过程

### 阶段 E：高价值动画与选择性 3D

目标：
只把最值得的概念做成真正拉开差异的视觉演示。

必须完成的事：

1. 只选 2 到 3 个最适合空间表达的主题
   推荐：
   - 圆运动空间演示
   - 坎离水火交通
   - 厥阴逆冲或六经层次

2. 为这 2 到 3 个主题做正式分镜和资产拆解

3. 决定每个主题是：
   - 高级 2D/2.5D
   - 还是 `react-three-fiber` 的 3D

4. 做性能与移动端验证

阶段产出：

1. 1 到 2 个高质量视觉样板
2. 项目的辨识度展示内容

验收标准：

1. 这些场景真的提升理解，不只是好看
2. 它们在演示和传播上能代表项目高度

### 阶段 F：病例推演与成品打磨

目标：
把项目从“内容产品”推进到“成品版本”。

必须完成的事：

1. 做 3 到 5 个病例推演
   让用户判断先手、后手、错误路线。

2. 把模块间路径收紧
   从基础到扶阳到病例，不再松散跳转。

3. 完成说明页
   项目介绍、学习说明、使用边界、非医疗建议声明。

4. 做一版发布检查
   性能、移动端、文案、路由、404、内容遗漏。

5. 产出展示版与学习版
   展示版给外部看，学习版给真实用户用。

阶段产出：

1. 可对外展示的正式版本
2. 可持续迭代的学习产品底盘

验收标准：

1. 外部用户第一次打开不会迷路
2. 项目可展示，也可持续学习

## 6. 建议的时间线

如果按一个中等强度节奏推进，可以按下面估算：

1. 阶段 A：2 到 3 周
2. 阶段 B：3 到 4 周
3. 阶段 C：2 到 3 周
4. 阶段 D：2 到 3 周
5. 阶段 E：3 到 5 周
6. 阶段 F：2 到 3 周

总计：

**大约 14 到 21 周**

如果是单人推进，按 4 到 6 个月看更现实。

## 7. 每周工作节奏

建议每周都按固定顺序推进，不要乱切：

1. 周前半做内容与结构
2. 周中做页面和交互
3. 周后半做测试与收口
4. 每周至少保留一个可演示结果

每周例行检查：

1. 这周新增了什么内容资产
2. 这周新增了什么学习路径
3. 这周新增了什么练习闭环
4. 这周有没有把原型变成可复用结构

## 8. 角色分工建议

如果你是一个人推进，可以按三种角色切换：

1. 内容编辑
   负责拆书、写课程、写题目、统一术语。

2. 产品设计
   负责模块结构、学习路径、练习规则、页面节奏。

3. 前端实现
   负责组件、交互、数据结构、页面、性能。

如果后面扩团队，最需要补的是：

1. 内容主编
2. 图表动效设计
3. 前端开发

## 9. 必须控制的风险

### 风险 1：过早沉迷 3D

解决：
3D 只在阶段 E 集中做，不提前吞噬主线资源。

### 风险 2：内容没拆完就疯狂做页面

解决：
先完成覆盖地图和模块脚本，再继续大规模扩页面。

### 风险 3：做了很多图，但没有学习闭环

解决：
每做一张重点图，都必须回答它接哪节课、哪道题、哪个错因。

### 风险 4：产品越来越像“演示站”，不像“学习站”

解决：
优先补题库、错题、进度、回看。

### 风险 5：两本书覆盖不均衡

解决：
做覆盖地图，逐章对照，不再只靠感觉。

## 10. 决策门

每个阶段结束都问这几个问题：

1. 用户能不能更容易学会，而不是只是更容易惊叹？
2. 内容是不是更完整了，而不是页面更多了？
3. 新做的东西是不是能复用，而不是一次性 demo？
4. 如果现在停止开发，当前版本能不能单独成立？

只要有两项答不上来，就不要急着开下一阶段。

## 11. 最建议的执行顺序

如果从今天开始，最合理的顺序是：

1. 先补 `内容覆盖地图`
2. 再补齐剩余 4 个模块脚本
3. 再升级练习题库与错题机制
4. 再做图表目录和图表台账
5. 再补学习进度页
6. 再挑 2 到 3 个主题做高质量动画或 3D
7. 最后做病例推演、发布检查和对外演示版

## 12. 对这个项目最重要的一句话

这个项目不是“把两本书做成会动的网页”，而是：

**把两本复杂中医书，重构成普通人真的能学进去的学习系统。**

只要你一直按这条主线推进，项目就不会跑偏。
